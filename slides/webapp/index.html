Module A: Agent Loop

AI Agent ทำงานเป็นวงจร โดยเริ่มจากการกำหนด Intent ให้ชัดเจน จากนั้นให้ข้อมูล Context สั่ง Action รับ Output แล้ว Validate ผลลัพธ์ — หากยังไม่ดีพอ ให้ Iterate โดยกลับไปปรับ Intent ใหม่แล้ววนซ้ำ ไม่ใช่แค่ retry คำสั่งเดิม

จาก Intent สู่ Prompt

Intent คือสิ่งที่คุณต้องการในหัว — ก่อนเปิด terminal เสมอ Prompt คือการแปล Intent นั้นให้ AI เข้าใจได้ชัดเจนโดยใช้ 5 กฎ

Intent (ในหัวคุณ)

"ต้องการสรุปยอดขาย Q1 เพื่อนำเสนอทีม — ต้องดูได้ทันที ไม่ยาวเกินไป และต้องมีตัวเลขยืนยัน"

→

Prompt (5 กฎ)

Role → senior analyst
Goal → สรุปยอดขาย Q1 จากไฟล์แนบ
Constraints → ภาษาไทย, ≤200 คำ
Output format → markdown table
Quality check → ทุก insight มีตัวเลขอ้างอิง

กฎข้อที่ขาดในหัวมักหลุดออกจาก prompt ด้วย → ผลลัพธ์เกินขอบเขตหรือ Validate ไม่ได้

Prompt Design Rules

5 กฎนี้คือสิ่งที่ต้องใส่ในทุก prompt เพื่อให้ Agent Loop ทำงานได้ครบวงจรตั้งแต่ Action จนถึง Validate

  • Role clarity — บอก AI ว่าเป็นใคร เช่น "คุณคือ senior analyst" ช่วยให้ตัดสินใจได้ถูกทิศทาง
  • Goal — บอกว่าต้องการทำอะไร เช่น "สรุปยอดขาย Q1" คือหัวใจของ prompt ขาดไม่ได้
  • Constraints — กำหนดขอบเขต เช่น ภาษา ความยาว หรือสิ่งที่ห้ามทำ ป้องกัน Output เกินขอบเขต
  • Output contract — ระบุรูปแบบผลลัพธ์ล่วงหน้า เช่น markdown table, JSON ทำให้ Validate ง่ายขึ้น
  • Quality checks — ตัวชี้วัดที่ใช้ใน Validate step เช่น "ต้องมีตัวเลขอ้างอิงทุก insight"

Prompt Design Rules — ตัวอย่าง

เปรียบเทียบ prompt ที่ ไม่มีโครงสร้าง กับ prompt ที่ใช้ 5 กฎ เพื่อให้เห็นความแตกต่างของผลลัพธ์

❌ Weak Prompt

สรุปยอดขายให้หน่อย

✅ Structured Prompt (5 Rules Applied)

Role: You are a senior sales analyst with 10 years of experience.
Goal: สรุปยอดขาย Q1 2026 จาก @sales.csv
Constraints: ภาษาไทย, ความยาวไม่เกิน 200 คำ, ห้ามเดาตัวเลขที่ไม่มีในข้อมูล.
Output format: markdown table with columns: Region | Revenue | Growth% | Key Insight
Quality check: ทุก insight ต้องอ้างอิงตัวเลขจากข้อมูลจริงเท่านั้น
  • Role clarity → "senior sales analyst" กำหนดมุมมองและระดับการวิเคราะห์
  • Goal → บอกว่าต้องการอะไร ไม่มีข้อนี้ AI ไม่รู้จะเริ่มต้น
  • Constraints → ภาษาไทย + ความยาว + ห้ามเดา ป้องกัน hallucination
  • Output contract → markdown table พร้อม column ที่ชัดเจน ทำให้ Validate ง่าย
  • Quality check → กฎตรวจสอบที่ฝังอยู่ใน prompt ทำให้ AI self-check ก่อนส่งผล

Safety Model

ก่อนสั่ง Agent ให้ทำงานที่มีผลกระทบสูง ต้องเข้าใจ 3 กลไกนี้เพื่อป้องกันความเสียหายที่แก้ไขไม่ได้

  • Approval gates — กำหนดจุดที่ต้องให้มนุษย์อนุมัติก่อน Agent จะดำเนินการต่อ เช่น การลบไฟล์ การส่งอีเมล หรือการ deploy โค้ด
  • Trust boundaries — ระบุขอบเขตที่ Agent มีสิทธิ์เข้าถึง เช่น อ่านได้แต่เขียนไม่ได้ หรือเข้าถึงได้เฉพาะ folder ที่กำหนด
  • Risky command awareness — รู้จักคำสั่งที่อันตราย เช่น rm -rf, DROP TABLE และตั้งกฎให้ Agent ขอยืนยันก่อนทุกครั้ง

Module B: Gemini CLI Core

Gemini CLI มี 2 โหมดหลัก — Interactive REPL สำหรับทดลองและ iterate และ One-shot mode สำหรับ automate งานใน script

  • Interactive REPL — พิมพ์ gemini เพื่อเข้าสู่โหมดสนทนา เหมาะสำหรับ iterate prompt หลายรอบและทดสอบ context injection
  • One-shot mode — ใช้ -p "..." เพื่อส่ง prompt เดี่ยวและรับผลลัพธ์ทันที เหมาะสำหรับใช้ใน shell script และ pipeline
gemini -p "Summarize top capabilities for beginners"

Essential CLI Flags

CommandWhat it does
geminiStart interactive REPL mode
gemini -p "..."One-shot prompt, exits after
gemini -p "..." > out.mdSave output to file
gemini -p "@file.md ..."Inject file as context
cat data.csv | gemini -p "..."Pipe content in
gemini --model gemini-2.0-flash -p "..."Specify model
gemini --helpList all flags

Module C: Content Automation

แทนที่จะเขียน content แยกทีละช่องทาง เราจะสร้าง brief เดียวแล้วให้ Agent แปลงออกมาเป็นหลาย format พร้อมกัน

One brief → many channels

  • Brief ที่ดี — ระบุ product/service, target audience, unique value proposition, และ CTA ที่ชัดเจน เพื่อให้ Agent สร้าง content ที่ consistent ทุกช่องทาง
  • Multi-channel output — จาก brief เดียว สร้างได้ทั้ง social caption, email draft, landing page copy ทั้งภาษาไทยและอังกฤษในคำสั่งเดียว
  • Version control — บันทึกเป็น content-pack.md และ content-pack-v2.md เพื่อ track การปรับปรุงและเปรียบเทียบผลลัพธ์

Content Quality Checklist

ใช้ checklist นี้ใน Validate step ของ Agent Loop เพื่อตรวจสอบ content ก่อน publish ทุกครั้ง

  • Message consistency — core message และ tone เหมือนกันทุกช่องทาง ไม่มีข้อมูลขัดแย้งกันระหว่าง social, email, และ landing page
  • Audience match — ภาษาและระดับรายละเอียดเหมาะกับกลุ่มเป้าหมาย เช่น technical terms สำหรับ developer แต่ plain language สำหรับ general public
  • Conversion clarity — CTA ชัดเจน มีเส้นตายหรือ urgency และผู้อ่านรู้ทันทีว่าต้องทำอะไรต่อไป
  • No unsupported claims — ทุก claim ที่มีตัวเลขหรือสถิติต้องอ้างอิงได้ ห้าม Agent สร้างข้อมูลที่ไม่มีในต้นฉบับ brief

Module D: Web App Integration Pattern

รูปแบบการเชื่อม AI เข้ากับ web app ที่ปลอดภัย — API key อยู่ server-side เสมอ ไม่เคย expose ไปที่ browser

Frontend → Backend → AI → Structured JSON

  • Frontend — รับ input จากผู้ใช้และส่ง request ไปที่ backend ของเราเอง ไม่เคย call AI API โดยตรงจาก browser เพื่อป้องกัน API key ถูกขโมย
  • Backend — Express server รับ request, validate input, แล้ว call Gemini API พร้อม API key ที่เก็บไว้ใน environment variable
  • Structured JSON — ระบุใน prompt ให้ AI ตอบกลับเป็น JSON เสมอ เพื่อให้ฝั่ง frontend parse และแสดงผลได้อย่างน่าเชื่อถือ

Integration Guardrails

3 กฎนี้ต้องปฏิบัติทุกครั้งที่นำ AI เข้า production — ละเลยข้อใดข้อหนึ่งอาจทำให้ app พัง หรือ API key ถูกขโมย

  • Keep keys server-side — เก็บ GEMINI_API_KEY ใน .env และโหลดผ่าน process.env เท่านั้น ห้าม hardcode หรือ commit ลง git repository
  • Add timeout handling — ตั้ง timeout สำหรับทุก AI call (แนะนำ 30 วินาที) และ handle กรณีที่ request ค้างหรือ model ไม่ตอบสนองด้วย fallback message
  • Return readable errors — เมื่อเกิด error ส่ง message ที่ user เข้าใจได้กลับไป ไม่ใช่ stack trace หรือ raw API error ที่อาจเปิดเผย internal structure

Module E: Data Insight Pattern

แทนที่จะอ่าน CSV ด้วยตาเปล่า เราจะใช้ Agent วิเคราะห์และสรุป insight ที่ผู้บริหารตัดสินใจได้ทันที

Raw CSV → trends → anomalies → recommendations

  • Trends — Agent หาแนวโน้มที่เกิดขึ้นต่อเนื่อง เช่น ยอดขายเพิ่มขึ้นทุกเดือน หรือ support ticket ลดลงหลัง product update
  • Anomalies — ระบุจุดผิดปกติที่ต้องตรวจสอบ เช่น ยอดขาย drop 40% ในสัปดาห์ที่ 3 หรือ ticket volume พุ่งสูงผิดปกติในหมวดหมู่ใดหมวดหมู่หนึ่ง
  • Recommendations — สรุปเป็น action items ที่ระบุ owner, timeline, และ KPI วัดผลได้ชัดเจน ไม่ใช่แค่ observation ลอยๆ

Manager-Ready Reporting

รายงานที่ดีต้องทำให้ผู้บริหารตัดสินใจได้ภายใน 2 นาที — ใช้ 3 องค์ประกอบนี้เป็น output contract ใน prompt

  • Concise summary — สรุปสถานการณ์รวมใน 3–5 bullet ไม่เกิน 2 บรรทัดต่อข้อ เน้น what happened และ why it matters
  • Quant-backed findings — ทุก finding ต้องอ้างอิงตัวเลขจาก data จริง เช่น "ยอดขาย Q1 เพิ่มขึ้น 23% YoY" ไม่ใช่ "ยอดขายดีขึ้น"
  • Prioritized actions — เรียง action items ตามผลกระทบและความเร่งด่วน แต่ละข้อระบุ owner, deadline, และ KPI ที่วัดความสำเร็จได้

Capstone Brief

Capstone ที่ดีคือ workflow ที่ทีมเอาไปใช้จริงได้ทันที — เตรียม 4 องค์ประกอบนี้ก่อนเริ่ม

  1. Problem statement — ระบุปัญหาที่ชัดเจนและวัดได้ เช่น "ทีม 5 คนใช้เวลา 2 ชั่วโมงต่อสัปดาห์เขียน report ด้วยมือ รูปแบบไม่สม่ำเสมอ"
  2. Prompt strategy — อธิบาย role, constraints, output contract, และ quality checks ที่ใช้ใน prompt พร้อมเหตุผลที่เลือก
  3. Output artifact — ผลลัพธ์ที่จับต้องได้ เช่น workflow table, content pack, หรือ insight report ที่บันทึกไว้ใน deliverables folder
  4. Validation approach — วิธีที่ใช้ตรวจสอบว่า output ตรงตาม success criteria ใช้ rubric, self-review prompt, หรือ peer review

Evaluation Rubric

Dimension1 — Weak3 — Good5 — Excellent
Problem framingUnclearSpecific taskClear + scoped + measurable
Prompt qualityGenericHas role + formatRole + constraints + output contract
Output utilityNot reusableUseful with editsImmediately reusable by team
Safety awarenessNot consideredMentioned riskValidation + fallback defined
CommunicationHard to followClear structureConcise, compelling, reproducible

Target score: 12 / 25 to pass  |  20+ = team-ready
